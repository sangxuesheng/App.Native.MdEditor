# Office 格式支持实施建议（面向当前项目）

## TL;DR
- **可行性**：
  - `docx/xlsx`：高可行 ✅
  - `pptx` 读取：中低可行 ⚠️（建议实验性）
- **优先级**：`docx > xlsx > pptx（实验性）`
- **原则**：不改现有核心能力路径，采用渐进式增强 + 可回滚开关。

---

## 一、项目现状与适配性结论
当前项目的 `fileFormats.js + App.jsx + server.js` 架构，天然适合扩展 Office 支持：
- 已有按扩展名分类与渲染策略；
- 已有统一文件读取 API 与权限校验（`resolveSafePath`）；
- 已有导出能力和 UI 扩展点。

**但需要注意**：`server.js` 已较大，继续堆逻辑会放大回归风险。建议先做结构隔离再上功能。

---

## 二、建议的目标边界（防止范围失控）
### 1) MVP 目标（第一阶段）
- 打开并预览 `.docx`（文本/基础结构）
- Markdown 导出 `.docx`（语义映射，不追求 CSS 级 1:1）
- 打开 `.xlsx`（限制 Sheet 数与行数）
- Markdown 表格导出 `.xlsx`（基础可用）

### 2) 非 MVP（延后）
- `.pptx` 高保真预览
- Office 复杂格式 1:1 还原

### 3) PPTX 策略
- 标记为实验性：`pptx-experimental`
- **MVP 不做预览**，仅显示文件信息与“暂不支持预览”提示
- 在 UI 明确“实验性/可能不完整”
- 后续如需文本提取，再单独立项评估 zip/XML 解析成本

## 三、架构与代码组织建议（必须先做）

## 1) 后端路由隔离
建议新增：
- `app/server/officeHandler.js`（解析逻辑、限制、错误映射）
- `app/server/officeTypes.js`（错误码、限制常量）

`server.js` 仅保留路由分发：
- 推荐：`/api/file/office/extract`
- 可扩展：`/api/file/office/metadata`

> 说明：保持 `/api/file` 现有行为不变，避免影响 md/text/image/pdf 现有链路。

## 2) 特性开关建议
不要只靠环境变量，建议接入现有设置系统（`/api/settings`）：
- `officeSupport.enabled`（默认 true）
- `officeSupport.pptxExperimental`（默认 false）

前端按设置动态渲染入口，便于热切换与回滚。

## 3) 前端按需加载依赖
- `docx/xlsx` 用动态导入（在导出时加载）
- 避免首屏包体积增大与初始化卡顿

## 四、技术选型建议（按当前项目约束）

### 1) DOCX
- 读取：`mammoth`（后端）
- 写入：`docx`（前端，动态导入）

导出策略：
- 走 Markdown 语义映射（标题、段落、列表、表格、代码块）
- 不做复杂 HTML/CSS 到 DOCX 的全量转换

### 2) XLSX
- 读取：`xlsx`（后端）
- 写入：`xlsx`（前端，动态导入）

读取限制：
- `maxSheets`、`maxRows`、`maxCellChars`
- 超限返回标准错误码

### 3) PPTX
- 读取：实验性（文本提取/元数据）
- 写入：可暂缓；若后续需要再引入 `pptxgenjs`

## 五、安全与性能硬限制（必须实现）

## 1) 文件大小限制
建议默认值：
- `docx`: 25MB
- `xlsx`: 15MB
- `pptx`: 10MB（实验性）

## 2) 统一错误码
- `OFFICE_TOO_LARGE`
- `OFFICE_PARSE_TIMEOUT`
- `OFFICE_FORMAT_UNSUPPORTED`
- `OFFICE_PARSE_ERROR`

## 3) 解析超时与隔离
- **MVP：主线程解析 + 超时控制（5~10s） + 降级返回**
- **Phase B：按环境能力启用 Worker Thread / 子进程**（不支持则自动降级主线程）
- 避免阻塞 Node 主线程影响其他 API

## 4) 压缩包安全（zip 容器）
- 校验魔数
- 限制解压后总大小、条目数、单文件大小
- 防 zip bomb（异常压缩比）

## 六、UI/UX 约束建议

## 1) 文件树展示
- `docx/xlsx` 正常图标
- `pptx-experimental` 显示实验性标记（颜色/tooltip）

## 2) 预览提示
Office 读取后显示提示：
- “当前为转换预览模式，可能存在格式损失”

## 3) 保存与导出文案
明确区分：
- “保存原文件”（如不支持则禁用）
- “导出为 docx/xlsx”
- “另存为 Markdown”

## 4) Office 文件布局策略（针对左右/仅编辑/仅预览）
为避免误导用户，Office 文件采用“按格式默认布局 + 用户可覆盖”的策略：

- `docx`：默认 **左右布局（编辑 + 预览）**
  - 原因：需要同时查看转换效果与文本编辑结果
- `xlsx`：默认 **仅预览**（表格视图）
  - 提供“切换到仅编辑（文本/JSON）”入口作为高级选项
- `pptx-experimental`：默认 **仅预览（信息态）**
  - 展示文件信息与能力提示，不提供可编辑承诺

交互约束：
- 自动布局切换仅在“首次打开该格式”触发，避免反复打断用户
- 用户手动切换布局后记忆偏好（按格式维度）
- 提供“锁定当前布局”选项，禁用自动布局策略

## 5) Office 文件可编辑能力边界（用户打开时）
为控制预期与回归风险，明确能力矩阵如下：

- `docx`：**可修改**（转换编辑）
  - 支持文本级编辑与导出新 `.docx`
  - 不承诺复杂样式无损回写
- `xlsx`：**可修改（基础）**
  - 支持基础单元格内容编辑与导出新 `.xlsx`
  - 复杂公式、样式、合并单元格等高保真能力延后
- `pptx-experimental`：**不可修改（MVP）**
  - 仅展示文件信息/提示，不提供编辑与预览承诺

建议在 UI 显示统一提示：
- “DOCX：支持转换预览与内容编辑，可导出为新 docx”
- “XLSX：支持基础表格编辑，可导出为新 xlsx”
- “PPTX：当前仅支持查看文件信息，暂不支持编辑”

## 七、分阶段实施计划（建议）

### Phase 0（1 天）— 重构准备
- 抽离 Office 处理模块
- 统一错误码与限制常量
- 接入配置开关（settings）

### Phase A（6~8 天）— MVP
- 格式常量扩展（含 `pptx-experimental`）
- docx 读取（后端）
- docx 导出（前端语义映射）
- xlsx 基础读取（限制 sheet/rows）
- 文件树图标与提示
- 主线程超时控制 + 大小限制 + 错误处理

### Phase B（4~6 天）— 稳定性与性能增强
- xlsx 导出（Markdown 表格基础映射）
- Worker/子进程能力评估与按环境启用
- 性能与内存优化（缓存、切换体验）

### Phase C（可选）— pptx 实验性深化
- 在明确收益前，维持“仅文件信息，不预览”
- 若继续投入，再评估文本提取实现与维护成本
- 配置开关控制可见性，灰度收集反馈

## 十、文件树高频切换场景专项优化（重点）

用户可能在文件树中频繁切换文件。该场景会同时影响文件树、编辑器与预览区渲染性能，需要专项治理。

### 1) 可能出现的性能影响
- 文件树节点全量重渲染，导致点击延迟、滚动抖动
- Monaco 模型反复销毁/重建，导致闪烁与卡顿
- Office 解析与预览更新阻塞主线程，造成帧率下降
- 缓存策略不当导致内存持续上涨

### 2) 文件树层优化（FileTree）
- 对树节点组件做 `React.memo`，仅让 active 路径与受影响分支重渲染
- 大目录启用虚拟化渲染（可基于现有 `react-window`）
- 搜索过滤与高亮计算使用 `useMemo`，避免每次切换全量重算
- 图标/格式判断结果做轻量缓存，减少重复字符串解析

### 3) 文件加载层优化（App）
- 引入 `AbortController` + 请求序号机制，确保“后发请求覆盖先发请求”
- 采用保守 LRU 缓存：默认最多 3 个文件，且总大小 ≤ 10MB（后续按设备能力放宽）
- 切换时先展示骨架屏/loading，再异步填充内容，降低感知延迟
- 对大文件触发降级策略（只读摘要/前 N 行），并给用户明确提示

### 4) 编辑器与预览层优化
- Monaco model 按文件路径复用，避免频繁重建
- docx/xlsx 解析走后端或 worker，避免前端主线程阻塞
- Office 预览更新使用分阶段渲染：先文本后结构化内容
- 导出依赖（`docx`/`xlsx`）动态加载，仅在导出流程触发

### 5) 建议性能验收指标（上线门槛）
- 常规文件切换响应时间 `< 100ms`
- 大文件切换首反馈（loading/skeleton）`< 200ms`
- 连续切换 30 次无明显掉帧（主观 + Performance 面板）
- 切换高峰后内存可回落，不出现持续爬升

## 十一、低侵入集成策略（按当前项目务实收敛）

为降低回归风险，Office 集成改为“最小文件增量 + 最少 App.jsx 改动”。

### 1) App.jsx 最小改动原则
- 保留现有 `switch-case` 主逻辑，不引入注册表重构
- 仅在末尾新增 Office 分支与渲染入口
- 控制在“最小接线改动”（目标 10~30 行，避免大规模重构）

### 2) 前端新增 2 个核心文件
- `app/ui/frontend/src/hooks/useOfficeHandler.js`：
  - Office 文件加载、超时、取消请求、错误映射、缓存管理
  - 对外仅暴露简洁方法（如 `loadOfficeFile`）
- `app/ui/frontend/src/components/OfficePreview.jsx`：
  - Office 预览展示（docx/xlsx）
  - `pptx-experimental` 仅展示“暂不支持预览”与文件信息

### 3) 导出链路务实实现
- `ExportDialog` 保持现状，仅补充 Office 导出选项与调用
- 导出依赖使用 Vite 兼容的动态导入：
  - `const { Document, Packer } = await import('docx')`
  - `const XLSX = await import('xlsx')`

### 4) 配置与回滚
- 接入 `/api/settings` 运行时开关（不改现有 settings 表结构）
- 出现问题可快速关闭 Office 能力，不影响 md/text/image/pdf 核心链路

### 5) 环境约束补充
- 离线优先：所有依赖本地打包，不依赖 CDN
- 飞牛 NAS：严格复用现有路径权限校验与绝对路径规则
- 微信浏览器：导出下载链路提供兼容提示与降级方案

## 十二、移动端专项优化（新增）

针对手机与微信浏览器场景，Office 能力必须采用“轻量渲染 + 可降级 + 可中断”策略，避免影响现有移动端体验。

### 1) 移动端性能策略
- 保守缓存：默认最多 3 个文件，且总缓存 ≤ 10MB
- 动态加载：`docx/xlsx` 仅在导出或相关功能触发时加载
- 请求治理：文件切换时启用防抖与 `AbortController` 取消旧请求
- 大文件降级：超阈值时仅显示摘要/前 N 行，并提示可切换到桌面端处理
- 首反馈优先：先显示 skeleton/loading，再异步填充内容

### 2) 移动端布局策略
- `docx`：移动端默认“仅预览”，用户可手动切换到编辑
- `xlsx`：移动端默认“仅预览（表格视图）”
- `pptx-experimental`：移动端仅显示文件信息与“暂不支持预览”提示
- 降低双栏使用频率，避免小屏下频繁重排与卡顿

### 3) 移动端交互策略
- 增大关键按钮触控区域，降低误触
- 解析与导出过程显示明确进度与状态文案
- 提供“取消加载/取消导出”能力
- 错误提示简短明确（超时、文件过大、格式限制）

### 4) 微信浏览器与离线兼容
- 导出下载链路提供微信环境兼容提示与降级方案
- 所有依赖本地打包，不依赖 CDN（离线优先）
- 失败时提供替代路径（如导出失败提示后续桌面端处理）

### 5) 移动端验收指标
- 常规文件切换响应时间 `< 100ms`
- 首次可感知反馈（loading/skeleton）`< 200ms`
- 内存增加控制 `< 10%`
- 重点场景交互目标：接近 60FPS

---

## 十三、最终建议
- 先做“可用”而非“完整还原”；
- 先把 `docx/xlsx` 跑通并稳定，再考虑 `pptx`；
- 坚持“新增能力独立模块 + 开关可回滚 + 核心链路零侵入”；
- 将“文件树高频切换性能”作为独立验收项，避免功能可用但体验退化；
- 采用“低侵入集成策略”，确保新增功能不继续扩大 `App.jsx` 复杂度；
- 将移动端体验作为单独质量门禁，避免桌面可用但手机端退化。

该路径可在保障现有功能稳定的前提下，稳妥扩展 Office 支持能力。

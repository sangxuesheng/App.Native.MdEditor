# 移动端预备改造清单

## 目标

这份清单用于指导当前桌面端为主的 Markdown 编辑器，逐步过渡到“未来可支持移动端/触控”的结构，而不是现在立刻全面开发移动端版本。

原则：

- 优先做“未来不返工”的底层准备，不急着做完整移动端界面。
- 优先处理输入模型、布局模型、弹窗滚动与可点击性。
- 尽量保持现有桌面体验不退化。
- 继续保持 `doXxx + handleXxx` 的交互分层，方便未来替换成长按、手势、底部菜单等移动端入口。

---

## 高优先级

这些文件建议在真正开始移动端准备时最先处理。

| 文件 | 未来移动端改造目标 | 建议方案 | 风险点 |
|---|---|---|---|
| `app/ui/frontend/src/components/Resizer.jsx` | 支持触控拖拽调整分栏 | 从 `mouse` 事件迁移到 `pointer` 事件，保留桌面拖拽逻辑；增加 `pointercancel` 处理 | 直接影响编辑器主布局，改动后若处理不好会造成桌面拖拽异常 |
| `app/ui/frontend/src/components/ElasticSlider.jsx` | 支持触控滑动调整数值 | 从 `mousemove/mouseup` 迁移到 `pointerdown/pointermove/pointerup`；增加触控命中区域 | 目前桌面行为简单稳定，迁移时要避免滑块跳值 |
| `app/ui/frontend/src/App.jsx` | 建立移动端布局切换基础 | 增加明确断点策略，规划桌面三栏/双栏与窄屏单栏或抽屉布局；不要先零散改 CSS | 这是布局总控文件，先改这里比到处修组件更有效，但也最容易带来回归 |
| `app/ui/frontend/src/components/FileTree.jsx` | 窄屏下文件树可用 | 后续改成抽屉/页签/全屏面板之一；保留当前树形逻辑，但不要假设永远并排显示 | 当前树节点、收藏、大纲、历史都塞在一个侧栏里，移动端如果仍沿用桌面结构会非常拥挤 |
| `app/ui/frontend/src/components/ImagePreview.jsx` | 支持触控缩放与更友好的窄屏预览 | 保留按钮缩放；后续补双指缩放/拖动；按钮区域加大点击热区 | 图片预览是移动端体验敏感区，触控手势与滚动冲突要谨慎处理 |
| `app/ui/frontend/src/components/ImageManagerDialog.jsx` | 从桌面弹窗过渡到移动端可用的全屏式图片管理 | 先把动作入口继续显式化；后续按“标签页 + 全屏页面”思路改，而不是硬压缩桌面弹窗 | 当前信息量大、标签多、图片库密度高，直接缩到手机宽度会明显难用 |

## 推荐实施顺序

建议按下面顺序推进，避免边做边返工：

1. `Resizer.jsx`、`ElasticSlider.jsx`
   先把底层输入模型从 `mouse` 思维过渡到 `pointer` 思维。
2. `App.jsx`
   明确移动端布局断点与面板呈现方式。
3. `FileTree.jsx`
   把文件树/收藏/大纲/历史在窄屏下的承载方式确定下来。
4. `ImagePreview.jsx`
   处理图片预览在触控场景下的缩放与按钮可点性。
5. `ImageManagerDialog.jsx`
   再决定图片管理是继续做弹窗，还是改为全屏页面式结构。
6. 其他弹窗与配置面板
   在整体布局定稿后做跟进适配。

## 开发前检查项

正式开始移动端准备前，建议先确认这些约束：

- 是否要支持手机浏览器访问，还是只考虑平板与触屏设备。
- 是否需要支持双指缩放、长按菜单、横竖屏切换。
- 是否允许移动端与桌面端界面结构不一致。
- 文件树、图片管理、导出配置这些“复杂面板”是否接受改成全屏页面式流程。

---

## 当前结论

当前项目最值得提前准备的，不是“马上做移动端 UI”，而是：

- 避免继续新增只支持鼠标的底层交互。
- 避免继续把复杂功能全部塞进桌面式弹窗。
- 继续保持业务动作和交互入口分层。

这样等后续真的开始做移动端时，改动会集中在布局与输入层，不会回头大面积重写业务逻辑。
